# Schedule Studio 畅想与优化路线

这份文档用于讨论 Schedule Studio 下一阶段可以怎么演进。它不是立即要全部实现的任务清单，而是一个产品和技术方向的草图。后续可以从这里挑选一个小主题，拆成具体实现计划。

## 当前项目是否适合使用

当前项目适合：

- 个人本地使用
- 学习 HTML、CSS、JavaScript、DOM、localStorage
- 管理轻量任务
- 练习项目结构拆分
- 练习 Git 小步提交

当前项目暂时不适合：

- 多设备同步
- 多人协作
- 长期保存重要日程
- 大量复杂重复日程
- 和真实日历软件双向同步
- 移动端深度使用

结论：

```text
当前版本适合当作“学习型日程工具”和“本地个人实验项目”。
如果要变成真正每天依赖的日程产品，还需要补数据可靠性、输入体验、同步能力和智能辅助。
```

## 当前数据保存方式评估

目前任务数据保存在浏览器 `localStorage` 中。

相关文件：

```text
assets/js/core/storage.js
assets/js/app.js
```

当前使用的 key：

```text
schedule-studio-data-v1
schedule-studio-lang-v1
schedule-studio-season-v1
```

### 优点

- 不需要服务器
- 实现简单
- 打开页面就能保存
- 很适合学习 JSON、localStorage、浏览器存储
- 数据不会上传，隐私性较好

### 缺点

- 只能保存在当前浏览器
- 换电脑、换浏览器后数据不同步
- 清理浏览器数据可能丢失任务
- 不适合大量数据
- 没有版本迁移机制
- 没有自动备份
- 没有冲突处理

### 短期改进

- 加“自动导出备份”提醒
- 加“上次保存时间”
- 加“数据统计面板”
- 加“导入前预览”
- 加“导入前备份当前数据”
- 给数据增加 `version` 字段

建议数据格式：

```json
{
  "version": 1,
  "updatedAt": "2026-05-27T10:00:00.000Z",
  "tasks": []
}
```

### 中期改进

可以从 `localStorage` 升级到 `IndexedDB`。

适合场景：

- 任务数量变多
- 需要保存更多结构化数据
- 需要离线优先
- 需要更稳定的浏览器端数据库

### 长期改进

如果要多设备同步，需要后端。

可选方案：

```text
Supabase      数据库、登录、实时同步，上手相对快
Firebase      前端友好，适合快速做云同步
自建 Node API  最灵活，但需要自己做账号、鉴权、数据库
GitHub Gist   适合极简个人同步，不适合正式产品
```

## 添加日程是否好用

当前添加日程是传统表单模式。

优点：

- 字段清楚
- 实现稳定
- 容易理解
- 数据结构可控

不足：

- 输入成本较高
- 每次都要手动选日期、时间、标签、优先级
- 不支持自然语言
- 不支持重复日程
- 不支持冲突提醒
- 不支持智能推荐时间
- 不支持从一句话生成任务

当前体验更像：

```text
表单录入工具
```

理想体验可以变成：

```text
一句话日程助手
```

例如输入：

```text
明天下午三点和 Alex 开产品评审，预计 1 小时，优先级高
```

系统自动解析成：

```json
{
  "title": "Product review with Alex",
  "titleAlt": "和 Alex 开产品评审",
  "date": "2026-05-28",
  "start": "15:00",
  "end": "16:00",
  "priority": "high",
  "tag": "studio",
  "notes": ""
}
```

## 智能日程添加方向

智能添加可以分三层做，不需要一开始就接大模型。

### 第一层：规则解析

先用本地规则实现简单解析。

支持：

```text
今天
明天
后天
下周一
上午九点
下午三点半
1小时
高优先级
学习 / 健康 / 工作 / 行政 / 旅行
```

优点：

- 不需要 API key
- 不花钱
- 响应快
- 可控
- 适合学习正则、日期解析、字符串处理

缺点：

- 语言变化多时会变复杂
- 容错能力有限
- 中文自然语言解析会比较麻烦

### 第二层：规则解析 + 确认面板

用户输入一句话后，系统先生成草稿，不直接保存。

流程：

```text
用户输入自然语言
系统解析成任务草稿
用户确认或修改
点击保存
```

这是一个很好的产品设计，因为它允许系统“不完全聪明”，但用户仍然可控。

### 第三层：接入大模型

大模型适合处理更自由的表达。

例如：

```text
这周五之前帮我安排三次跑步，每次 45 分钟，避开晚上
```

模型可以输出结构化 JSON：

```json
{
  "tasks": [
    {
      "title": "Run",
      "titleAlt": "跑步",
      "date": "2026-05-28",
      "start": "07:30",
      "end": "08:15",
      "priority": "low",
      "tag": "health",
      "notes": "Auto-planned from natural language"
    }
  ]
}
```

## 是否可以接入大模型

可以，但不建议直接在前端暴露 API key。

### 不推荐方式

```text
浏览器前端直接调用大模型 API
```

原因：

- API key 会暴露
- 容易被滥用
- 无法控制成本
- 无法做服务端日志和限流

### 推荐方式

增加一个后端代理。

结构：

```text
浏览器
  -> /api/parse-schedule
    -> 后端携带 API key 调用大模型
      -> 返回结构化任务 JSON
```

后端可以做：

- API key 保护
- 请求限流
- JSON schema 校验
- 日志记录
- 成本控制
- 用户权限

### 大模型适合做什么

- 自然语言解析成任务
- 生成任务标题和中文/英文标题
- 判断标签
- 判断优先级
- 估算时长
- 拆分复杂计划
- 检查日程冲突
- 根据空闲时间推荐安排
- 根据天气调整户外活动

### 大模型不应该直接做什么

- 不应直接无确认保存任务
- 不应绕过用户确认
- 不应把模型输出当作一定正确
- 不应在没有校验的情况下写入本地数据

建议原则：

```text
AI 负责生成草稿
用户负责确认
程序负责校验
```

## 天气 API 是否可以加入

可以，而且和日程产品很搭。

天气适合影响这些场景：

- 跑步
- 旅行
- 通勤
- 户外活动
- 运动计划
- 拍摄安排
- 节假日出行

### 可以显示什么

```text
今日天气
未来 7 天天气
温度
降雨概率
风力
空气质量
日出日落
恶劣天气提醒
```

### 可以和日程怎么结合

```text
如果明天下雨，提醒户外任务
如果温度过高，建议调整跑步时间
如果空气质量差，提醒减少户外运动
旅行计划前显示目的地天气
```

### 前端直接调用天气 API 可以吗

有些天气 API 可以前端直接调用，但要看 API key 是否允许暴露。

更稳妥方式仍然是后端代理：

```text
浏览器
  -> /api/weather?city=Shanghai
    -> 后端调用天气 API
      -> 返回简化天气数据
```

### 天气数据缓存

天气不需要每次刷新都请求。

可以缓存：

```text
当前天气：15-30 分钟
每日预报：1-3 小时
```

这样可以减少请求次数，提高速度。

## 技术路线建议

### 阶段 1：把当前项目打磨顺

目标：让项目作为本地日程工具更舒服。

建议任务：

- 优化添加任务表单
- 加快速添加栏
- 加数据备份提醒
- 加任务搜索
- 加重复日程的基础模型
- 加更清晰的空状态

### 阶段 2：增强数据层

目标：让数据更可靠。

建议任务：

- 给数据加 `version`
- 导入时做 schema 校验
- 导入前自动备份
- 增加数据迁移函数
- 评估 IndexedDB

### 阶段 3：智能添加

目标：从“填表”进化到“一句话添加”。

建议任务：

- 先做规则解析器
- 设计智能添加确认面板
- 加解析失败提示
- 支持中英文自然语言
- 再考虑接入大模型

### 阶段 4：天气和上下文

目标：让日程知道外部环境。

建议任务：

- 加城市设置
- 接入天气 API
- 加天气缓存
- 在今日面板显示天气
- 对户外任务显示天气提醒

### 阶段 5：后端和同步

目标：从本地工具变成跨设备产品。

建议任务：

- 选择后端方案
- 增加用户登录
- 任务云同步
- 大模型 API 代理
- 天气 API 代理

## 推荐下一步

最推荐先做：

```text
智能添加的第一版：本地规则解析 + 确认面板
```

原因：

- 不需要后端
- 不需要 API key
- 对用户体验提升明显
- 技术难度适中
- 很适合学习字符串解析、表单预填、数据校验

第一版目标：

```text
用户输入：“明天下午3点学习JavaScript 1小时”
系统生成任务草稿
用户确认后保存
```

这一步完成后，再接大模型会更稳，因为我们已经有了：

```text
任务草稿格式
确认面板
数据校验
保存流程
```

大模型只是把“解析器”从规则版升级成 AI 版，而不是重写整个添加流程。
